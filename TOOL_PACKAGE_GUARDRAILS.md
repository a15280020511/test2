# 临时工具插头边界

## 核心规则

上游工具包由上游团队维护。`test2` 不建立第二套包维护平台，只维护最薄的本地集成边界。

所有非控制核心的第三方工具必须采用任务级临时插头：

```text
需要时安装
→ 只在当前任务临时环境中运行
→ 输出日志与结果证据
→ 无论成功、失败或超时都清理
→ Runner结束后整体销毁
```

不用时，第三方依赖不得留在DeepSeek永久控制环境中。

## 永久核心允许保留

- Python标准库；
- GitHub Workflow和GitHub API调用；
- 官方DeepSeek HTTP客户端；
- JSON、HMAC、任务状态、去重、取消、强制取消、日志和Artifact控制；
- 插件清单验证和临时环境执行器。

## 必须插件化

- Microsoft Agent Framework；
- OpenRouter SDK；
- 专业统计、仿真、优化、预测、空间、因果和其他第三方分析工具；
- 任何不是DeepSeek/GitHub控制面启动所必需的包。

## 插件组成

每个插件只保留：

```text
plugins/<name>/plugin.json
plugins/<name>/requirements.txt
必要的薄适配器
最小契约测试
```

不得建立：

- 常驻插件服务器；
- 插件数据库；
- 自动fork和长期维护上游源码；
- 后台持续更新服务；
- 无边界自动安装任意包；
- 多套重复的包管理系统。

## 权限与隔离

插件只能获得当前任务必需的：

- 网络权限；
- 临时文件目录；
- 明确传入的非秘密环境变量；
- 受控结果输出目录。

插件不得：

- 读取或打印Secrets；
- 修改 `.git/`；
- 修改测试、控制工作流、运行结果分支或其他任务文件；
- 获得仓库写权限，除非经过DeepSeek REPAIR和独立验证；
- 静默扩展依赖或执行未在清单允许的模块。

## 资源边界

每个插件清单必须规定：

- 允许的模块；
- 允许的操作；
- 安装超时；
- 运行超时；
- `cleanup: always`。

模型插件还必须服从用户批准的预算：

- 最大费用；
- 预计费用区间；
- 最大模型调用次数；
- 每次最大输出token。

## 故障责任

DeepSeek Steward负责：

- 插件清单错误；
- 薄适配器错误；
- 仓库内依赖冲突；
- Workflow集成错误；
- 上游接口变化导致的本地兼容问题；
- 插件清理失败；
- 预算闸门和审计错误。

DeepSeek Steward不负责：

- 上游项目路线图；
- 上游发布和安全维护；
- 重写上游包；
- 默认建立永久fork；
- 为了追新版本而不断修改仓库。

上游更新本身不构成仓库故障。只有当前任务确实无法正确运行、安全问题必须处理或新能力被明确要求时，才修复最小本地边界。

## 选择原则

网页 GPT 根据具体任务选择最少必要插件。DeepSeek ASSIST负责审计插件需求和预算，REVIEW负责检查结果和预算，REPAIR只修复本地集成边界。

> 小核心、临时插头、强隔离、用完销毁、上游维护归上游。
